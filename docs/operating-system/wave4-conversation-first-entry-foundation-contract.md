# Wave 4 Conversation-First Entry Foundation Contract

## Purpose

`W4-AGENT-005A` exists to replace the canonical Nexus entry order without destroying the product intelligence already built underneath.

The required shift is:

- from `create -> onboarding -> understanding -> action`
- to `conversation -> hidden understanding -> visible direction -> downstream action`

This contract does not close the whole pivot.
It locks the exact foundation the later implementation tasks must follow.

## Why This Is A New Canonical Task

This work does not belong to `W4-GEN-002`.

`W4-GEN-002` is about later product mutation from feedback and prior outcomes.
This pivot is about first-run entry architecture, hidden bootstrap, conversation durability, route ownership, and downstream continuity.

It belongs in the post-Wave-4 learning and intake continuation lane because it extends:

- adaptive intake
- family-aware questioning
- correction and clarification behavior
- live conversation tone and pacing

It does not justify a new lane.

## Current Reality Audit

Repository and live-flow reality still center Nexus around staged setup:

1. `/` still resolves to the `create` route.
2. first-run flow still depends on `projectName` and `visionText`.
3. onboarding session creation still happens after explicit create state.
4. restore still rebuilds conversation truth from multiple stores.
5. downstream surfaces still assume the user willingly walked through staged onboarding.

That means Nexus still behaves like:

- visible project creation
- visible onboarding
- then continuation

which is now canonically wrong for the product direction.

## Canonical Product Decision

Nexus entry must now be governed by this rule:

`onboarding is a hidden orchestration layer, not a visible setup ritual`

Visible entry should become:

1. one conversation-first front door
2. one first free-language message
3. automatic hidden project/session/brief bootstrap
4. one useful partner response
5. one clarifying question only when the ambiguity changes the product

## Replacement-First Boundary

The current `create + onboarding` front door may not be removed or degraded until all are true:

1. the replacement conversation-first front door exists
2. it is visually coherent with the Nexus shell
3. it is live-verified
4. it is continuity-safe
5. it is strong enough to become the default without lowering product quality

During migration:

- old entry remains available as fallback or secondary utility
- new entry may not demote old entry before replacement proof exists

## Target Architecture

### 1. Route Ownership

The default visible entry route must become the conversation-first front door.

The old routes must be reassigned as:

- `create`: fallback or secondary manual path
- `onboarding`: hidden orchestration surface or explicit recovery route
- `understanding`: downstream reflection surface attached to the same conversation origin

The route flip may happen only after replacement proof and structural design proof both exist.

### 2. Identity Model

Nexus must distinguish:

- temporary conversation identity
- persistent session identity
- persistent project identity
- hidden brief identity
- downstream timeline/history identity

The first user message may start with temporary conversation identity,
but it must deterministically promote into one persistent session/project chain without forking truth on refresh or retry.

### 3. Hidden Bootstrap Model

After the first real message:

1. create or resolve a session
2. infer family and likely structure
3. create or resolve a hidden brief
4. create or resolve project identity in the background
5. reply immediately with product momentum

This must not depend on visible `projectName` or `visionText` fields.

### 4. Canonical Truth Store Rule

Conversation-first entry requires one canonical conversation truth.

The canonical truth may not be split across:

- local draft inputs
- local route state
- transient summary snapshots
- project knowledge mirrors
- QA merge residue

Other stores may exist only as derived caches from the same canonical conversation/session truth.

### 5. Durability Rule

No user turn may be lost.

This includes:

- first idea
- clarification reply
- retry during delay
- refresh during in-flight processing
- route transition
- restore after degraded runtime state

### 6. Idempotency Rule

Repeated clicks or retries may never create duplicate canonical user turns.

The visible client may show pending state,
but the canonical backend/session truth must dedupe the same intent into one user turn.

### 7. Restore And State-Isolation Rule

Refresh must restore the exact same conversation/project truth.

A new project entry may never inherit:

- old summary
- old missing-items
- old family truth
- old preview truth
- old QA scenario merge
- old project understanding

### 8. Conversation Quality Rule

The new entry path must preserve and strengthen:

- family reasoning
- challenge mode
- dual-audience modeling
- paraphrased follow-up questions instead of raw answer pasting
- pre-class idea-specific ambiguity rather than generic missing-items

Conversation-first entry may not become hidden wizard behavior.

## Structural Design Threshold

This pivot crosses the structural threshold because it changes:

- top-level route ownership
- front-door hierarchy
- shell expectations for first-run
- relationship between entry, onboarding, and downstream surfaces

Therefore broad implementation may not proceed truthfully until a Figma-backed structural pass defines:

1. the replacement front door
2. entry shell hierarchy
3. progressive reveal of companion/understanding/downstream surfaces
4. fallback location of old create/onboarding routes
5. migration-safe shell behavior during the transition

## Required Canonical Decomposition

The pivot must now execute through these tasks:

1. `W4-AGENT-005A`
   - define the conversation-first entry foundation contract and route ownership
2. `W4-AGENT-005B`
   - run the Figma-backed primary entry shell structural pass
3. `W4-AGENT-005C`
   - implement the conversation-first front door with hidden project/session/brief bootstrap
4. `W4-AGENT-005D`
   - harden durability, idempotency, restore, and stale-state isolation on the new entry path
5. `W4-AGENT-005E`
   - connect the conversation-first origin truthfully into Understanding and downstream continuity
6. `W4-GEN-003`
   - only later, transition the new entry path into entry-to-live-skeleton continuity

This ordering is mandatory.

## Live Proof Expectations For Later Tasks

Later implementation must prove all of the following on the real Nexus product:

1. first-run begins in conversation
2. first AI reply appears without staged setup
3. project/session/bootstrap happen automatically
4. refresh preserves the same active thread
5. retry does not duplicate turns
6. delay does not lose turns
7. weak answers are challenged
8. dual-audience products are modeled truthfully
9. old project truth cannot leak into a new conversation
10. the replacement front door feels like Nexus rather than a generic AI console

## Closure Boundary For W4-AGENT-005A

`W4-AGENT-005A` closes only when:

1. this contract exists canonically
2. the task decomposition is written back into canonical state
3. `W4-GEN-002` is demoted back behind the new entry tasks

This task does not require the replacement front door to be live yet.
