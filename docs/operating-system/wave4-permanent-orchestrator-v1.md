## Wave 4 Execution Doctrine

### 0. New Shell Migration Doctrine

Wave 4 is now responsible not only for the internal `Minimum Believable Nexus` engine,
but for carrying that engine into the new visible Nexus shell.

That means execution must now preserve three simultaneous truths:

- preserve existing canonical engines
- remove the old orchestration-first visible shell
- build the new Nexus product surfaces and close the first truthful vertical slice

No later task may bypass:
- shell removal
- surface pass
- shell-to-engine bridge
- first vertical slice

if those remain unresolved release blockers in the implementation task map.

### 1. Minimum Believable Nexus First

Wave 4 execution is not organized around:
- GTM first
- owner shell first
- internal engine depth first

It is organized around making Nexus truthfully usable as:
- a class-aware product creation engine
- a visible build system
- a local workspace
- a release-aware product loop
- a continuation engine

### 2. Visible Build Progression Doctrine

If Nexus claims it is building a product,
the user must see the product building.

No execution task may become `trueGreen` if the result exists only as:
- internal state
- orchestration logic
- artifact text
- proof text
- hidden runtime state

### 3. Class-Aware Generation Doctrine

Wave 4 must treat product class as a primary execution axis.

The class must influence:
- bootstrap
- preview frame
- workspace shape
- runtime direction
- shell behavior
- release path
- continuation logic

### 4. Build-Before-UI Prohibition

Build-before-UI is forbidden.

No meaningful backend/core/runtime capability may advance alone if its user-facing surface, route, continuity, or workspace manifestation is not progressing with it.

### 5. Coupled Backend-To-Surface Doctrine

Meaningful capability work must advance together across:
- backend
- orchestration
- routing
- persistence
- UI
- build surface
- proof/artifact/timeline continuity
- release path
- continuation path

### 6. Workspace Reality Doctrine

Wave 4 must move Nexus toward:
- split workspace
- live build surface
- simulator or preview frames
- local-first continuity
- Electron or desktop-shell truth where required

### 7. Release And Continuation Doctrine

Wave 4 does not stop at “generated output exists”.

Execution must push toward:
- releaseable state
- visible runtime direction
- visible release path
- visible post-release continuation

### 8. External Product Intelligence Doctrine

Wave 4 must learn from external systems actively,
but every imported pattern must enter through:
- canonical lane ownership
- canonical task structure
- validation rules
- continuity rules
- design/system integration

### 9. Figma-Backed Structural Evolution Doctrine

If a capability changes structural product surfaces,
execution must stop for a Figma-backed design pass before implementation continues.

Figma integration is no longer a future placeholder.
It is a real Wave 4 execution capability.

But it is not the product creation engine.

Figma may assist:
- design exploration
- layout planning
- structural visual evolution
- design-system continuity

Figma may not replace:
- Nexus class resolution
- Nexus product bootstrap
- Nexus product generation
- Nexus runtime selection
- Nexus release shaping
- Nexus continuation logic

The orchestrator must know:
- when to stop for a Figma pass
- when to create or update a Figma file
- when structural UI change requires design evolution before implementation
- when Figma output becomes part of the canonical execution flow

Structural changes include:
- split workspace
- build surfaces
- simulator layout
- Electron shell
- release surfaces
- orchestration UI
- continuation flows

### 10. Product Reality Over Metadata Doctrine

Wave 4 success is measured by:
- what the user sees
- what the user can continue
- what the product can release
- how coherent the workspace feels

Not by:
- metadata richness
- orchestration verbosity
- hidden architecture elegance

## Wave 4 Delegation Model

### Core Model

Wave 4 uses:
- one main orchestrator
- many bounded execution workers where useful

This is not peer-to-peer autonomy.

### Delegation Is Allowed When

- a class-specific slice can be isolated
- a design slice can be isolated
- a bounded verification problem can run in parallel
- a targeted implementation investigation can run without redefining task order
- a focused external research subtask can accelerate planning or closure

### Delegation Is Forbidden When

- the work changes canonical task ordering
- the work changes lane structure
- the work changes validation gates
- the work decides `trueGreen`
- the work writes canonical state independently
- the work promotes new lanes without orchestrator judgment

### What Execution Workers May Do

- bounded research
- bounded architecture analysis
- bounded design prep
- bounded Figma prompt preparation
- bounded Figma file updates when already scoped and approved
- bounded implementation
- bounded verification
- evidence gathering

### What Only The Main Orchestrator May Do

- select the next canonical task
- reinterpret planning artifacts
- approve external pattern integration
- approve Figma-triggered structural evolution
- approve structural design direction from Figma output
- decide closure status
- decide rerun discipline
- decide lane/task promotion
- write canonical truth
- decide wave transitions

### Delegation Write-Back Rule

Workers may return:
- findings
- evidence
- diffs
- test results
- candidate fixes
- Figma prompts
- Figma file links
- Figma design deltas

Only the main orchestrator may:
- classify final task state
- update canonical lane truth
- declare `trueGreen`

## Wave 4 Figma Evolution Rules

Structural UI implementation is blocked until a Figma pass exists when the change affects:
- workspace structure
- build surfaces
- simulator layout
- Electron shell
- release surfaces
- orchestration UI
- continuation flows

Figma is now an executable Wave 4 capability.

### Figma Trigger Threshold Rule

The change is small enough for inline evolution if ALL of the following are true:
- no layout hierarchy changes
- no navigation model changes
- no new workspace region or panel is introduced
- no split-workspace behavior changes
- no simulator/preview frame logic changes
- no Electron/local shell frame changes
- no release/continuation surface structure changes
- no component-system divergence risk is introduced

If any of the above is false, a structural Figma pass is mandatory.

### Figma Execution Rule

When the threshold is met for a structural pass, the orchestrator must:
1. stop implementation expansion
2. create or update the relevant Figma file
3. define the design contract
4. link the Figma artifact back to the active canonical task
5. only then allow structural implementation to continue

### Where Figma Is Required

Figma is required only when Nexus itself needs structural design evolution in:
- workspace architecture
- split workspace behavior
- build surfaces
- simulator or preview frame architecture
- Electron / local shell frame
- orchestration UI structure
- release surfaces
- continuation surfaces

### Where Figma Is Forbidden

Figma is forbidden as a substitute for:
- user-product generation
- Nexus bootstrap logic
- class resolution
- runtime selection
- release shaping
- continuation logic
- the internal product creation engine itself

The required Figma pass must define:
- layout
- hierarchy
- interaction model
- component structure
- navigation behavior
- split workspace behavior
- build/simulator behavior
- systematic integration rules

Figma output becomes part of canonical execution flow when:
- it defines a structural contract for an active Wave 4 task
- the orchestrator approves the direction
- the task write-back references the Figma artifact as required input

Patchwork UI is forbidden.
