## Wave 4 Founder Field-Test Task Intake Prompt

Use this prompt whenever a live founder journey, competitor pass, or adversarial product run exposes real Nexus failures that are stronger than the current canonical closure claims.

### Prompt

Continue from the canonical Nexus system of record.

You are not doing generic backlog grooming.
You are converting a live founder field test into canonical Wave 4 execution truth.

Source of truth:
- the live Nexus product and its visible routes
- the recorded founder journey
- the canonical Wave 4 ledger

Your job:
1. extract only repository-relevant product failures from the report
2. separate root causes from downstream symptoms
3. decide whether the report:
   - opens follow-on fix tasks
   - reorders the next canonical task
   - or proves an earlier closure was too narrow and must be reopened by a new repair chain
4. write tasks in dependency order, not emotional order
5. preempt any downstream generation or mutation task if upstream identity, hidden-brief, class, or continuity truth is poisoned

Hard rules:
- do not treat severe live findings as optional improvements
- do not leave `nextCanonicalExecutionTask` pointing at downstream work if upstream truth is invalid
- do not collapse multiple root causes into one vague repair task
- do not let a founder report about wrong-product output skip directly to copy polish
- do not preserve prior `trueGreen` confidence as a reason to avoid opening a fix chain

Dependency ordering rules:
1. project identity isolation comes before all other fixes
2. hidden brief / summary / placeholder flush comes before classifier tuning
3. classifier humility and family-confidence gating come before class-specific questioning
4. multi-actor modeling comes before downstream skeleton truth
5. route-state architecture repair comes before claiming restore/privacy/continuity are solved
6. companion truthfulness comes after the canonical brief is trustworthy
7. convergence and first-skeleton escape hatches come after the upstream truth chain is repaired
8. feedback-driven mutation may not proceed while conversation-first understanding is still contaminated

Required output shape:
- one short reopener statement
- one ordered canonical task chain
- exact reason each task is ahead of the next
- one write-back that changes the next canonical task if needed

Failure patterns that must open fix work immediately:
- reused project identity across fresh conversations
- stale hidden brief or stale family truth leaking into a new idea
- explicit user correction failing to flush the old brief
- confidently wrong family assertions
- dual-audience collapse into a false single actor
- full serialized conversation state in the URL as route truth
- companion panels showing fabricated understanding
- unconverging onboarding that blocks first skeleton visibility

Success condition:
The resulting canonical order makes Nexus safer, truer, and more founder-reliable before any later mutation or optimization work continues.
