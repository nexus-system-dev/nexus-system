# Nexus Canonical Product System

Date: 2026-05-26
Status: Canonical working draft
Scope: Consolidated product, system, and operating truth for the next Nexus rebuild

## 1. Core Identity

Nexus is a conversation-first AI builder that understands products deeply enough to build a believable live product quickly, then continue evolving it over time.

Nexus is not:
- a chatbot
- a template gallery
- an orchestration dashboard
- an IDE clone
- a plugin marketplace with AI around it

Nexus is:
- a product-building intelligence
- one coherent system
- cloud-canonical
- desktop-augmented
- build-first
- momentum-first

Core law:
- Nexus is autonomous in creation, but not in sovereignty.

## 2. Canonical User Promise

The sharpest product promise is:
- Nexus turns an idea into a living, evolving product, not just a pretty screen or starter code.

The user should feel:
- Nexus understood what I am trying to build.
- The product is appearing in front of me.
- What I say changes the product.
- Nexus remembers and continues with me.

## 4. Canonical Product Entity

The canonical entity in Nexus is the Product Graph.

The Product Graph contains:
- idea
- conversation-derived intent
- understood product truth
- actors
- flows
- surfaces
- data model
- design system
- release state
- growth learnings
- history

Important distinctions:
- project = container
- surface = canonical product surface inside product truth
- artifact = visible implementation/expression of a surface at a moment in time
- Product Graph = canonical truth

Core law:
- There is one product truth.
- All screens are views over that truth.

## 5. Source of Truth Architecture

All major surfaces are layers over the same Product Graph:
- conversation
- understanding
- build
- history
- release
- growth
- share
- studio

Truth precedence:
- conversation = intent
- committed Product Graph = current active truth
- artifact = expression of truth, not truth ownership
- release = published frozen snapshot of truth
- history = record of truth evolution
- manual edit = proposed or applied truth mutation

Canonical truth is:
- the latest approved or applied Product Graph state

Canonical representations:
- Current Product Graph = active truth now
- Current Artifact / Build = what is currently built from active truth
- Preview = what is being shown for work/review, not truth ownership
- Released Truth = immutable snapshot of Product Graph + artifacts + verification at release time

Artifact state semantics:
- proposed
- building
- applied
- verified
- committed
- released

Rule:
- artifact that is not committed is a candidate, not canonical truth
- committed Product Graph remains the living truth

## 8. Build Contract

Build is the heart of Nexus.

Build surface law:
- Build surface = persistent agent conversation rail + live artifact/build canvas in the same workspace.
- The user should feel: “I am talking to Nexus, and Nexus is building the product in front of me.”
- Discovery chat does not disappear. It compresses into the persistent right-side agent rail while the live build canvas opens beside it.
- The conversation does not disappear after discovery; it becomes the working rail that guides, explains, changes, verifies, and releases.
- The artifact/build canvas is not a detached preview; it is the live product surface being shaped by the agent and user.
- The user must experience one Nexus agent across discovery, build, mutation, verification, share, growth, package, and release. Internal agent boundaries may exist for engineering truth, but they must not appear as separate user-facing agents or reduce what the user can ask from the active rail.
- A user request in the active Nexus rail must either change the visible product and the relevant saved product truth, or show a precise blocker. A reply alone is not a product-building action.
- Requests to create assets, screens, actions, fields, flows, backend operations, package files, tests, growth drafts, share states, or release checks must route through the relevant internal capability while remaining one continuous Nexus action to the user.
- This paired workspace remains the primary work surface from first skeleton through release.
- Motion must be gentle, futuristic, respectful, and reduced-motion safe.

Always visible:
- conversation with Nexus
- live artifact surface
- human progress state
- fast way to change things

Build must feel like:
- a creative live workspace

Not:
- a chat window with a tiny preview
- a standalone chat that later hands the user to a disconnected builder
- a standalone artifact screen with no persistent agent conversation
- an orchestration console
- a classic IDE

First 5 seconds truth:
- Nexus is already building with me.
- This is a real product surface.
- I can still change direction.

Manual choice between “build now” and “clarify more” appears only when ambiguity is real and consequential.

## 17. Approval and Autonomy Boundary

Nexus does automatically:
- small safe edits
- sensible defaults
- continuity
- safe retries
- non-destructive improvements

Nexus suggests but does not auto-apply:
- architecture changes
- platform shifts
- major UX redesigns
- growth directions
- pricing/business suggestions

Nexus requires approval for:
- deployment
- publishing
- provider connections
- filesystem access
- local code execution
- destructive changes
- core flow replacement
- payment/realtime/security changes

Nexus never auto-does:
- public publishing
- silent integration connection
- permission escalation
- irreversible destructive acts
- strategic decisions on behalf of the user

## 18. Trust, Failure, and Recovery

Fast trust is built on:
- strong first skeleton
- visible progress
- real correction handling
- continuity

Failure recovery ladder:
1. preserve current work
2. return to last stable state
3. isolate failure
4. safe retry
5. safe fork if needed
6. ask user if high impact

Recovery must feel like:
- a safety net
- not restore roulette

Dead ends are real when momentum disappears across multiple rounds.
In dead ends, Nexus should:
- frame the blockage
- reduce scope
- offer 2 to 3 bounded paths
- keep visible movement alive

- Billing / Plans
- Permissions / Access Scopes

Canonical core law:
- the central entity is Product Graph
- not conversation
- not artifact
- not route state

### 37.2 Canonical persisted data

Must be persisted as canonical:
- Product Graph
- project identity
- branch identity
- artifact states
- surfaces

- data model
- design system
- decisions
- releases
- permissions

### 37.5 Anti-spaghetti structural rules

Structural rules:
- Product Graph is the central source of truth
- conversation stores intent, not final product truth
- artifact represents built state, not canonical truth ownership
- release does not silently rewrite Product Graph
- URL state is never canonical truth
- every important state is scoped to project / branch / surface
- there is no hidden shared project truth across products
- each service owns its data boundary clearly
- derived data must be rebuildable
- canonical data must be versioned
- meaningful changes should pass through explicit mutation/change records

### 37.6 What must never happen

Never allow:
- duplicated product truth across services
- conversation truth drifting away from product truth
- artifact drift
- URL-state-driven backend truth
- global project state masquerading as local truth
- reused hidden brief across projects
- schema drift without migration discipline
- caches becoming truth
- summaries replacing Product Graph
- release pointing at non-canonical artifact state
- cross-project memory contamination
- one giant backend blob without ownership boundaries
- temporary bridge architecture becoming permanent core architecture

## 38. Resilience / Backup / Recovery Ops

Nexus must be built on the assumption that things will break.

### 38.1 Operational failure classes

Resilience must cover:
- DB corruption
- failed migrations
- accidental deletion
- corrupted Product Graphs
- broken releases
- provider outages
- runtime crashes
- sync drift
- asset loss
- branch corruption
- stale deployments
- rollback failures
- local/cloud divergence
- preview fleet failures
- release misconfiguration
- memory corruption
- broken integrations
- partial writes

### 38.2 What must be backed up

Backups must exist for:
- Product Graphs
- branches
- releases
- artifacts
- design systems
- assets
- memory snapshots
- project history
- configs
- permissions
- collaboration state
- audit trails
- runtime metadata
- deployment metadata
- verification records
- integration mappings
- billing / identity state

Not every cache or transient runtime state needs backup.

### 38.3 Recovery strategy

Recovery modes:
- point-in-time restore
- branch restore
- release rollback
- safe replay
- partial recovery
- disaster mode

Disaster mode should:
- freeze safe state
- isolate damage
- restore last stable truth
- preserve user work

### 38.4 Automatic vs manual recovery ops

Automatic:
- retries
- safe rollback
- runtime reroute
- provider failover
- snapshot creation
- integrity checks
- preview rebuilds
- non-destructive restores

Manual / owner-gated:
- destructive restores
- cross-project recovery
- migration rollback
- release rollback in production
- mass recovery
- security incidents
- secret compromise
- org-wide restore
- branch merge recovery

Core law:
- what can be safely restored can be automated
- what can rewrite truth needs a human gate

### 38.5 Resilience without ops overload

The right resilience layer is:
- automatic snapshots
- strong isolation
- simple rollback
- clear restore flows
- internal observability
- a few strong guarantees

Not:
- giant ops consoles
- manual babysitting
- recovery bureaucracy

### 38.6 What must never happen

Never allow:
- backup theater
- snapshots that do not restore cleanly
- silent corruption
- restore roulette
- partial-truth recovery
- hidden data loss
- rollback chaos
- provider outage cascading into project corruption
- sync disasters without recovery paths
- fake redundancy
- irrecoverable product truth

Core resilience law:
- user trust must not depend on luck.

## 41. Canonical Mutation Model

Every meaningful change in Nexus must pass through one canonical mutation model, regardless of where it starts.

### 41.1 Mutation entry points

Mutation may begin from:
- conversation
- selection + command
- direct edit
- drag / reorder
- import
- screenshot / reference
- collaborator feedback
- release feedback
- growth suggestion
- branch compare
- verification finding
- recovery action

Core rule:
- all mutation sources converge into the same canonical mutation system
- not separate pipelines by interaction type

### 41.2 Canonical mutation stages

Canonical order:
1. Intent / Proposal
2. Interpretation
3. Impact Analysis
4. Approval if needed
5. Application
6. Verification
7. Commit to Product Truth
8. Checkpoint / History Update

Definitions:
- Intent / Proposal = what the user or Nexus wants to change
- Interpretation = what that change actually means in product terms
- Impact Analysis = what moves if this changes
- Approval = user gate for high-impact mutation
- Application = execute the mutation
- Verification = confirm meaningful integrity after change
- Commit = write the mutation into canonical Product Graph truth
- Checkpoint / History = preserve evolution trail where appropriate

### 41.3 Proposal vs applied mutation

Mutation stays a proposal when it is:
- a suggestion
- ambiguous
- high-impact and pending approval
- a possible direction
- a comparison state

Mutation becomes applied when:
- the user approves it
or
- it is safe enough for Nexus to apply automatically
and
- it has been executed into the product state

Canonical distinctions:
- intent = desire
- suggestion = proposed direction
- planned change = understood but not yet applied
- applied mutation = change committed into Product Graph truth

### 41.4 Required checks before meaningful mutation

Before meaningful mutation, Nexus must evaluate:
- blast radius
- affected surfaces
- affected actors
- release implications
- branch implications
- rollback safety
- consistency with Product Graph
- permissions / security impact
- runtime impact
- design consistency
- growth impact
- shared system impact

Core rule:
- Nexus must understand what else moves when this changes.

### 41.5 Post-mutation updates

After a successful mutation, the system must update:
- Product Graph
- artifacts
- surfaces
- history
- memory
- branches
- release readiness
- verification state
- growth assumptions
- previews
- snapshots
- collaboration state
- retrieval context

Core rule:
- never leave the system in half-old, half-new truth.

### 41.6 User-facing feel

Mutation must not feel like:
- a pipeline
- a workflow engine
- a commit ritual

It should feel like:
- Nexus understood the change
- the product changed

Sometimes:
- immediate
- quiet
- lightly highlighted

Sometimes:
- briefly framed
- when impact is meaningful

The user should feel:
- I am shaping the product with Nexus

Not:
- I am operating a mutation system

### 41.7 What must never happen

Never allow:
- hidden rewrites
- accidental truth replacement
- silent destructive changes
- mutations without rollback safety
- stale branches mutating current truth
- suggestions silently becoming truth
- partial updates
- inconsistent artifacts
- mutation loops without grounding
- two truths at once
- conversation promising one thing while build shows another
- orchestration chaos

Core law:
- mutation must feel visible, understandable, and coherent inside one living product.

## 44. Temporal Product State Philosophy

Nexus must understand that a product does not exist only as versions, but as multiple truth states across time. These states must stay coherent without competing for active reality.

### 44.1 Truth states Nexus must recognize

Nexus should recognize:
- Current Truth
- Intended Truth
- Experimental Truth
- Released Truth
- Deprecated Truth
- Archived Truth
- Blocked Truth
- Branch Truth

### 44.2 What defines each truth state

Current Truth:
- the active truth the product is currently built from

Intended Truth:
- a decided direction or mutation not fully applied yet

Experimental Truth:
- a direction under active test that is not yet the main product truth

Released Truth:
- an immutable snapshot of Product Graph + artifacts + verification that has actually been shipped:
  - preview
  - beta
  - public
  - production

Deprecated Truth:
- something once valid but no longer a basis for future building

Archived Truth:
- historical truth retained for traceability

Blocked Truth:
- a direction that cannot currently progress

Branch Truth:
- truth that exists only inside a particular branch or path

Abandoned Truth:
- a direction that was explored but is no longer part of the active future

### 44.3 How truth states must coexist

Core law:
- at every moment there must be one and only one Current Truth

All other truth states may:
- diverge
- wait
- remain historical
- remain experimental

But they must not compete with the active current reality.

The user must never be confused between:
- what is live
- what is planned
- what is experimental
- what is historical

### 44.4 What should be visible vs implicit vs hidden

Visible:
- current build
- released versions
- active branch
- major experiments
- meaningful deprecation warnings

Implicit:
- intended changes
- planned mutations
- future direction

Hidden:
- internal temporal state transitions
- orchestration queues
- abandoned internal branches
- stale recovery states
- low-level temporal machinery

Core rule:
- the user should not see a temporal state machine

### 44.5 Truth state transitions

Key transitions:
- intended → current
- current → released
- released → deprecated
- experimental → merged
- branch truth → abandoned

Transitions must be:
- reversible when feasible
- traceable
- understandable

Immutable temporal targets:
- releases
- checkpoints
- snapshots
- verification records
- exported builds
- published artifacts
- release metadata
- rollback targets

Rule:
- if a released state changes, it becomes a new release
- not a mutated old release

Not:
- temporal chaos
- silent state replacement

### 44.6 What must never happen

Never allow:
- multiple current truths
- stale truth resurfacing as active truth
- released truth drifting away from current build truth without clarity
- abandoned branches influencing active product
- hidden experimental truth leaking into release
- unclear active direction
- rollback causing temporal corruption
- “what is actually live?” confusion

Core law:
- Nexus must feel like one living product evolving coherently through time, not multiple competing realities.

## 50. Preview / Sandbox / Deploy Plumbing

Nexus must move products through preview, sandbox, and release plumbing in a way that feels simple to the user while remaining strictly isolated and operationally safe underneath.

### 50.1 What Preview is

Preview is:
- a live state for viewing and working with the product
- good enough to inspect
- good enough to interact with
- good enough to review UX and flows
- good enough to share for feedback

Preview is not:
- production
- release
- canonical truth itself
- a full unsafe execution environment

Core law:
- preview should feel like the product is alive enough to work with

### 50.2 What Sandbox is

Sandbox is:
- a safe, isolated execution environment

Use sandbox when Nexus needs to:
- run code
- test a dependency
- test an integration
- validate runtime behavior
- process imports
- execute anything not yet trusted

Sandbox must be:
- isolated
- limited
- scoped
- disposable
- non-canonical unless promoted through controlled mutation/release paths

Core distinction:
- Preview = live product experience
- Sandbox = safe experimental execution space

### 50.3 When something moves from Preview to Release Plumbing

A product state should move toward release plumbing when:
- it is stable enough
- key flows work
- basic verification has passed
- intent is clear
- architecture is not actively collapsing
- release target is clear
- no critical blockers remain unresolved

Core law:
- the transition happens when the product stops being only a direction and becomes something that could plausibly be shipped

### 50.4 How Deploy Plumbing should work

Deploy plumbing should be:
- mostly hidden
- automated where safe
- reversible
- policy-aware

It should include:
- packaging
- environment preparation
- runtime configuration
- deploy target preparation
- verification gates
- promotion between environments
- rollback hooks
- release snapshots

But:
- without turning Nexus into a DevOps console

### 50.5 User-facing language

User-facing meaning should be product-first:

Preview:
- you can already see and work with this

Sandbox:
- this is being tested safely

Release-ready:
- this is stable enough to send out

Never expose by default:
- container language
- pipeline jargon
- infra topology
- runtime plumbing details

### 50.6 Isolation boundaries

Must isolate across preview, sandbox, and release:
- data
- secrets
- runtimes
- providers
- branches
- release state
- environments
- assets
- auth
- integrations
- logs
- memory scopes

Never allow:
- sandbox touching production data by default
- preview using release secrets
- branch preview contaminating active release

### 50.7 What must never happen

Never allow:
- fake previews
- previews that do not reflect the real product state
- sandbox theater
- release confusion
- shared environments without isolation
- preview mutating production
- deploy chaos
- invisible release state
- branch/runtime contamination
- “it worked in preview but nowhere else” as the normal case
- DevOps UX taking over the product

Core law:
- the user must never feel like they are managing infrastructure; they should feel Nexus knows how to move a product from idea to live state to real release.

## 53. Project Health / Entropy Model

Nexus must detect not only sudden failures, but also slow product decay and structural entropy over time.

### 53.1 Slow degradation signals

Nexus should detect:
- branch sprawl
- too many abandoned directions
- repeated rollback
- stale releases
- artifact drift
- duplicated flows
- duplicated logic
- growing UI clutter
- growing permission complexity
- unresolved contradictions
- too many surfaces
- feature sprawl
- temporary systems becoming permanent by inertia
- verification degradation
- release hesitation
- slow build momentum
- declining consistency
- growing override chaos
- increasing recovery frequency

Core signal:
- the product is not broken yet, but it is starting to become messy

### 53.2 Healthy evolution vs experimentation vs decay vs risk

Healthy evolution:
- complexity with clear value
- branches that converge
- releases that keep moving
- coherent mutations
- flows that strengthen
- experimentation with clear direction

Messy experimentation:
- many experiments
- but bounded
- reversible
- non-contaminating to current truth

Slow decay:
- features without ownership
- stale branches
- duplicated flows
- growing confusion
- overrides everywhere
- unclear direction
- increasing friction

Actual product risk:
- nobody trusts current truth
- releases feel unsafe
- rollback becomes routine
- no one knows what is canonical
- mutations frequently break things
- build momentum disappears

### 53.3 What Nexus should do when a project becomes messy

Nexus should not panic or flood the user with warnings.

It should:
- suggest cleanup
- propose consolidation
- highlight risky complexity
- suggest refactor when useful
- freeze risky paths when necessary

Examples:
- there are three similar flows here that may be worth consolidating
- this permissions layer has become significantly more complex since the last release

Core rule:
- cleanup is guidance, not punishment

### 53.4 What should be visible vs implicit vs internal

Visible:
- risky release state
- meaningful duplicated flows
- stale experiments
- unresolved contradictions
- unstable branches
- verification degradation

Implicit:
- light cleanup suggestions
- refactor nudges
- organizational hints

Owner-only / internal:
- entropy scoring
- drift metrics
- mutation instability signals
- trust-risk calculations
- system degradation analytics

Core rule:
- users should not see health telemetry systems

### 53.5 Preventing management sludge

Never make health feel like:
- scores everywhere
- Jira-style risk boards
- warning spam
- anti-experimentation policing

Health should appear:
- contextually
- only when it matters
- in human language

Core rule:
- Nexus protects experimentation
- it intervenes against sustained chaos, not healthy exploration

### 53.6 What must never happen

Never allow:
- nagging system behavior
- entropy paranoia
- warning spam
- anti-experimentation culture
- AI over-cleaning products blindly
- forced simplification without product reasoning
- health bureaucracy
- management sludge
- project-score obsession
- every branch treated as risk
- every complexity treated as bad

Core law:
- project health in Nexus is not a compliance score; it is a mechanism for protecting clarity, momentum, and trust over time.

## 54. Semantic Diff / Change Explanation Philosophy

Nexus must explain change in product language, only when the change has real meaning.

### 54.1 Which changes deserve explanation

Nexus should explain changes that carry:
- product meaning
- real impact
- direction change
- behavior change
- structure change

Examples:
- structural changes
- workflow changes
- actor changes
- release-impacting changes
- permission / auth changes
- design system changes
- branch merge results
- architecture shifts
- growth-driven mutations
- release rollback
- cross-surface impact

It should not fully explain:
- every spacing tweak
- every tiny rename
- every small reorder

### 54.2 Explanation structure

Change should be explained in product language:

What changed:
- the permissions system now distinguishes between customers and staff

Why:
- so the admin dashboard is not open to every user

What was affected:
- this affects the app, the dashboard, and the next release

What it means now:
- customer onboarding and staff onboarding are now separate

Core rule:
- explanation is in product terms, not system terms

### 54.3 When Nexus must explain vs when it may stay quiet

Must explain when there is:
- behavior change
- flow change
- actor change
- cross-surface impact
- release impact
- risky mutation
- architecture change
- branch merge
- rollback
- tradeoff-bearing change

May stay quiet when there is:
- small polish
- spacing
- visual cleanup
- local copy tweak
- small reorder
- safe refinement

In those cases, enough may be:
- highlight
- animation
- visual update
- tiny note

### 54.4 Preventing explanation from becoming a log

Nexus must not narrate every mutation.

Never explain in phrases like:
- updated artifact
- mutation applied
- rebuild complete

Instead:
- explain only changes with meaning
- keep explanations short
- keep framing product-centric
- use visuals where possible

Preferred supporting forms:
- highlight
- compare
- before / after

Not:
- patch notes culture
- internal mutation narration

### 54.5 What the user should understand from change explanation

From each meaningful explanation, the user should understand:
- what really changed
- why it matters
- what was affected
- whether it is safe
- whether it is reversible
- how it moves the product forward

Core law:
- change explanation should reinforce that the product evolves coherently, not randomly

### 54.6 What must never happen

Never allow:
- commit commentary everywhere
- mutation narration
- orchestration logs
- AI over-explaining
- patch-note product behavior
- system-dump language
- technical diffs for normal users
- explaining obvious tiny changes
- “AI changed 14 nodes”
- endless change feeds

Core law:
- Nexus must not feel like a system that documents every change; it must feel like a living product builder that explains only what truly matters.
